## Why

Pages generated by `od_generate_design` are visually inconsistent across a single project — each call produces an internally coherent but mutually different palette, type ramp, and component vocabulary, because nothing pins a shared visual contract between turns. The daemon already exposes a `designSystemId` field on every project but this MCP never reads or writes it, so multi-page projects drift on every regenerate.

This change introduces a project-scoped, machine-readable **design system artifact** (tokens + components + layout primitives) that every subsequent page generation must consume and adhere to. It eliminates cross-page drift by construction (prompt injection) with a static lint backstop, while remaining fully opt-in so existing projects regress nothing.

## What Changes

- **NEW MCP tool `od_generate_design_system`** — BYOK-driven generation with a charter focused on tokens + components; emits a single `design-system.html` containing three layered `<style>` blocks (`#od-tokens`, `#od-components`, `#od-layout`), a JSON manifest (`<script type="application/json" id="od-design-system-manifest">`), and a human-reviewable component gallery in `<body>`. Fixed `kind: 'design-system'`.
- **NEW MCP tool `od_extract_design_system`** — pure function (no network) that parses a `design-system.html` and returns `{ manifest, tokensCss, componentsCss, layoutCss, version }`. Exposed for both internal reuse and agent introspection.
- **NEW MCP tool `od_update_design_system`** — patches an existing system; supports two modes: (i) BYOK semantic patch (e.g. "add a destructive button variant"), (ii) deterministic JSON-delta patch. Bumps the `data-od-version` attribute on `<html>` on every successful update.
- **MODIFIED `od_generate_design`** — when `projectId` resolves a project with a `designSystemId`, the MCP auto-fetches the referenced `design-system.html`, extracts the JSON + CSS, and prepends a "Design System Contract" block to the prompt (wrapper-injection in the MCP handler; vendored `composeSystemPrompt` is NOT modified). New optional arg `designSystemMode: 'strict' | 'advisory' | 'off'` — defaults to `'strict'` when a system is linked, `'off'` otherwise.
- **MODIFIED `od_lint_artifact`** — new optional input `designSystemHtml?: string`. When provided, runs additional static checks emitting finding codes DS001 (missing required `<style>` block), DS002 (off-palette color), DS003 (undocumented component class), DS004 (new `--color-*` / `--space-*` custom property), DS005 (component CSS not byte-identical to system's). An HTML comment `<!-- od-lint-ignore-next-line -->` SHALL suppress the next finding.
- **MODIFIED `od_compose_brief`** — new optional input `designSystemSummary?: string` that gets formatted into the Turn 3 prompt verbatim under a `### Design System` heading.
- **No schema changes** to `od_create_project` / `od_update_project`. The existing `designSystemId` field is the link; documentation only clarifies the convention (`designSystemId` points at the filename saved via `od_save_project_file`; the convention is `design-system.html`).
- **CHANGELOG + README + skills** updated to document the new opt-in workflow and the three new tools.

## Capabilities

### New Capabilities
- `design-system`: design-system artifact lifecycle (generate, extract, update) plus the constraints on its shape and the project-link convention.

### Modified Capabilities
- `tools`: requirement deltas for `od_generate_design` (auto-inject linked design system + `designSystemMode` arg), `od_lint_artifact` (new `designSystemHtml` arg + DS001–DS005 findings + `od-lint-ignore-next-line` escape hatch), and `od_compose_brief` (new optional `designSystemSummary` field).

## Impact

- **Code (new):** `src/tools/generate-design-system.ts`, `src/tools/extract-design-system.ts`, `src/tools/update-design-system.ts`, `src/types/design-system.ts` (Zod schemas + parsed-document type).
- **Code (modified):** `src/tools/generate-design.ts` (wrapper-injection branch + new `designSystemMode` arg), `src/tools/lint-artifact.ts` (new arg + DS001–DS005 rules), `src/tools/compose-brief.ts` (new optional field), `src/tools/index.ts` (register 3 new tools), `src/server.ts` (tool count assertion bumps from 10 → 13).
- **Vendor:** `vendor/od-contracts/` is NOT modified. The Design System Contract is injected by the MCP wrapper, not by `composeSystemPrompt`.
- **Tests:** new unit tests under `src/__tests__/` for extractor, manifest schema, prompt-injection, and each DS-finding code; new integration scenarios covering generate-system → save → link → generate-page → lint with `designSystemMode: 'strict'`.
- **Docs:** `README.md` gains a "Design-System-First Workflow" section and 3 new rows in the tool table; `.opencode/skills/od-workflow/SKILL.md` updated to make the new flow the default multi-page recommendation; `.opencode/skills/open-design-mcp/SKILL.md` adds the three new tool entries.
- **Release:** v0.17.0 (semver-minor, additive, fully backward-compatible). `CHANGELOG.md` entry explicitly calls out opt-in nature.

## Non-Goals

- **No `od_apply_design_system`** auto-repair tool (rewriting near-miss hex colors etc.) — deferred to v0.18.
- **No `od_regenerate_pages`** mass-rerender tool — deferred to v0.18 to avoid catastrophic regression on adoption.
- **No daemon-side schema changes.** The link is via the daemon's existing `designSystemId` field; storage uses the existing `od_save_project_file` endpoint.
- **No multi-system-per-project** support. One design system per project for v0.17.
- **No theming / dark-mode variants.** A single canonical system per project; theme tokens are a future change.
- **No modifications to vendored `composeSystemPrompt`.** All injection happens in the MCP wrapper layer.
